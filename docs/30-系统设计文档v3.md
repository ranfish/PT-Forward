## 一、项目概述与架构决策

### 1.1 项目背景与定位

**项目名称**: PT-Forward
**定位**: 综合性 PT 管理工具，支持多站点、多下载器、刷流/转发/辅种
**目标用户**: 个人/小团队 PT 爱好者
**部署环境**: NAS / VPS / Docker
**技术栈**: Go (后端) + Vue3 (前端)

### 1.2 核心功能模块

| # | 模块 | docs/31 章节说明 |
|---|------|-----------------|
| 1 | **RSS 订阅引擎** | §2/§6：独立 Go 包，EventHandler 接口，输出 TorrentEvent |
| 2 | **刷流引擎** | §14/§18：H&R 保护 + 做种策略 + SeedingEngine 接口 |
| 3 | **种子分发路由** | §9：TorrentDispatcher，基于角色路由（seeding/download/source/reseed） |
| 4 | **辅种引擎** | §15：E-1 手动/E-2 自动双模式，三层匹配 + 决策引擎 |
| 5 | **发布流水线** | §21：PublishPipeline 21 步，从源站抓取到目标站上传 |
| 6 | **站点管理** | §13：Site 统一模型 + SiteAdapter 框架适配 + CookieCloud |
| 7 | **下载器管理** | §1/§7：DownloaderClient 接口（25 方法）+ 四种角色 |
| 8 | **通知系统** | §16：NotifyService + ChannelNotifier（Telegram/邮件/Webhook） |

### 1.3 架构决策记录

#### 决策 1: 松耦合单体

- **结论**: 标准单体应用，单二进制文件部署
- **策略**: 接口隔离 + 依赖注入，后续可拆分
- **docs/31 参考**: §1.3

#### 决策 2: 数据库选型

- **结论**: **SQLite 为首要数据库**，GORM 抽象层可切换 MySQL
- **理由**: 零配置、适合个人 NAS/VPS 环境、GORM 统一抽象
- **docs/31 参考**: §33.1.41-51（11 张 GORM 表定义）+ §33.1.59（site_config_overrides）

#### 决策 3: 缓存策略

- **结论**: **内存 LRU + SQLite 持久化**，不使用 Redis/MySQL 缓存层
- **理由**: 替代 vertex 的 Redis 依赖，单机部署无需额外组件
- **docs/31 参考**: §2.1

#### 决策 4: 下载器角色模式

- **结论**: 四种角色（seeding/download/source/reseed），**publish 不是角色，是订阅级功能开关**
- **docs/31 参考**: §7（角色定义）、§9.5（postPush 路由逻辑）

#### 决策 5: 推送模式

- **结论**: **全程 AddFromFile**，移除 PushMode（URL 推送）
- **理由**: RSS 模块已下载种子数据，直接用 AddFromFile 注入
- **docs/31 参考**: §2.7

#### 决策 6: 并发模型

- **结论**: 7 个有界 goroutine 池（ADR-05）
- **池**: extractor(CPU×2), datasource-query(20), decision-engine(CPU×4), injector(5), site-sync(15), notification(5), hash-compute(CPU)
- **docs/31 参考**: ADR-05

#### 决策 8: 站点配置更新机制

- **结论**: `go:embed` YAML 只读默认值 + DB 字段级行覆盖 + 运行时合并
- **理由**: 彻底消除升级合并冲突；单二进制部署；支持字段级来源追踪和"恢复默认"
- **docs/31 参考**: §19.2（分层模型）、§19.7（存储方式）、§33.1.59（SiteConfigOverride）

#### 决策 9: 优雅关闭与中断恢复

- **结论**: 三阶段 grace period 关闭（默认 60s）+ 发布流水线步骤级断点恢复
- **理由**: 更新程序时在途发布任务可安全排空；步骤级断点避免重启后重复上传；其他模块天然幂等无需额外设计
- **docs/31 参考**: §33.39（优雅关闭与中断恢复专题）、§33.1.53（LastCompletedStep 字段）

## 三、五种辅种策略分析

> 本章保留原始分析，作为参考来源的历史文档。实际实现以 docs/31 §15 为准。

| 策略 | 准确率 | 覆盖率 | 数据源 | 隐私性 | 性能 |
|------|--------|--------|--------|--------|------|
| **pieces_hash 精确匹配** | 99.9% | 40-60% | 本地解析 | ⭐⭐⭐⭐⭐ | ⚡⚡⚡ |
| **IYUU API 匹配** | 95% | 60-80% | 云端 API | ⭐⭐ | ⚡⚡ |
| **ContentFingerprint** | 85% | 70-90% | 本地计算 | ⭐⭐⭐⭐⭐ | ⚡⚡⚡ |
| **文件树对比** | 90% | 50-70% | 本地/站点 | ⭐⭐⭐⭐ | ⚡⚡ |
| **info_hash 直接匹配** | 100% | 5-10% | 站点 API | ⭐⭐⭐ | ⚡⚡⚡⚡ |

### 3.1 pieces_hash 精确匹配（来自 reseed-puppy-php）

**原理**: pieces_hash 是种子文件 `info.pieces` 字段的 SHA1 哈希。即使不同站点使用不同的 announce URL，只要内容相同，pieces_hash 就相同。NexusPHP 1.8.5+ 提供 pieces_hash 查询接口。

**优势**: 准确率极高、完全本地、可批量查询。
**劣势**: 需要站点支持该接口。

### 3.2 IYUU API 匹配（来自 iyuuplus-dev）

**原理**: 上传本地做种的 info_hash 列表到 IYUU 服务器，在其数据库中查找匹配。覆盖率 60-80%，但有 API 限流（200 个/批次）。

> **docs/31 实际实现**: §33.2.7 IYUUService 接口（6 方法），§33.1.15 IYUUReseedResult/IYUUTarget/IYUUCandidate。

### 3.3 ContentFingerprint 匹配（来自 Graft）

**原理**: 通过 TotalSize + FileCount + LargestFileSize + FilesHash 多维度匹配。

> **docs/31 实际实现**: §33.1.33 ContentFingerprint（12 字段 GORM 模型），§33.2.22 FingerprintRepository（5 方法）。

### 3.4 文件树对比（来自 Reseed-backend）

**原理**: 比较种子文件路径和大小，允许容差。

> **docs/31 实际实现**: 自动化辅种**仅使用 STRICT 模式**（路径+大小精确匹配）。FLEXIBLE/PARTIAL 保留为手动审核模式。

### 3.5 info_hash 直接匹配

**原理**: 直接比较 info_hash。准确率 100% 但覆盖率极低（5-10%）。

| 不匹配 | `DecisionSizeMismatch` | 大小不匹配 |
| 不匹配 | `DecisionFileTreeMismatch` | 文件树不匹配 |
| 不匹配 | `DecisionFuzzySizeMismatch` | 模糊大小不匹配 |
| 不匹配 | `DecisionPartialSizeMismatch` | 部分大小不匹配 |
| 元数据 | `DecisionReleaseGroupMismatch` | 发布组不匹配 |

| 元数据 | `DecisionSourceMismatch` | 来源不匹配 |

| 已存在 | `DecisionSameInfoHash` | 相同 InfoHash |

### 4.3 核心数据模型

> **docs/31 参考**: §15.3（ReseedTask）、§33.1.33（ContentFingerprint）、§33.1.20（ReseedDecision）

**ReseedTask 核心字段**:
- `SourceSiteIDs string` / `TargetSiteIDs string`（源站/目标站分离）
- `EngineMode string`（"e1_manual" / "e2_auto"）
- `MatchMethods string`（JSON: ["pieces_hash", "iyuu", "fingerprint"]）
- `TargetSiteExcludes` / `ReleaseGroupExcludes` / `CategoryExcludes` / `TitleKeywordExcludes`（4 维黑名单）
- `Version int`（乐观锁）
- `MaxInjectionsPerRun` / `InjectionIntervalS` / `InjectionJitterS`（注入节流）
- `MaxRetries int` / `RetryIntervalH int`（失败重试）

**ReseedMatch 核心字段**:
- `SourceSite` / `SourceTorrentID` + `TargetSite` / `TargetTorrentID`
- `ClientID string`（多客户端并行路由）
- `DecisionType string` / `DecisionDetail string`
- `Status string`（matched/injecting/injected/failed/skipped）
- `RetryCount int` / `NextRetryAt` / `FailReason`
- `CachedTorrentData []byte`（暂存种子数据）
- **唯一索引**: `UNIQUE(client_id, source_site, source_torrent_id, target_site, target_torrent_id)`

## 五、数据模型设计

### 5.3 数据类表

#### content_fingerprints（内容指纹表，§33.1.33）

> GORM 模型，12 字段，替代旧版 torrent_hashes + content_fingerprints 双表设计。

#### site_config_overrides（站点配置覆盖表，§33.1.59，Sprint 89 决策 #232）

用户通过 Web UI 对站点配置的字段级覆盖，合并时 DB override 优先于 go:embed 默认值。

核心字段: ID, SiteName, FieldPath（点分路径如 `publish.mappings.type.category.movie`）, FieldValue（JSON 字符串）, Source（web_ui/yaml_import/api）
**唯一约束**: `UNIQUE(site_name, field_path)`

### 5.4 已移除的表（与旧版差异）

| 旧表 | 原因 | 替代方案 |

| `rules` | 模块级过滤替代 | RSSFilterRule + ReseedTask 内联过滤 + SeedingClientConfig |

## 八、Web 前端设计（概要）

> **完整设计见 docs/31 §34（§34.1-§34.15）**

### 8.1 技术栈

| 层面 | 选型 |
|------|------|
| 框架 | Vue 3 + TypeScript + Composition API |
| 组件库 | Ant Design Vue 4.x |
| 状态管理 | Pinia + Axios |
| 图表 | ECharts 5 |
| 构建 | Vite 5 |
| 部署 | Go embed 单文件 |
| 实时通信 | WebSocket (gorilla/websocket) |
| i18n | vue-i18n 9（第一阶段仅中文） |

### 8.2 API 设计

- RESTful JSON，URL 前缀 `/api/v1/`
- 统一响应信封 `{code, message, data}`
- 114 个端点，14 个模块
- Offset-based 分页 `?page=1&size=20`
- 必须认证 + JWT (HS256) 双 Token (Access 30min + Refresh 7d)
- 单用户模型：固定用户名 `admin`，启动时自动生成 8 位随机密码输出到 stdout
- 密码强度：≥8 字符，含大小写+数字+特殊符号
- 登录锁定：同 IP 连续失败 5 次锁定 5 分钟
- 忘记密码：CLI `--reset-password` 重置

### 8.3 页面结构

- 22 个页面，双布局（空白 + 主布局）
- 侧边栏 3 分组：资源管理 / 任务管理 / 系统
- 10 个 WebSocket 订阅通道，22 种事件类型
