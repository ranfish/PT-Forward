## 一、项目概述与架构决策

### 1.1 项目背景与定位

**项目名称**: PT-Forward  
**版本**: v3.0 (最终版)  
**定位**: 综合性PT管理工具，支持多站点、多下载器、刷流/转发/辅种引擎  
**目标用户**: 个人/小团队PT爱好者  
**部署环境**: NAS / VPS / Docker  
**技术栈**: Go (后端) + Vue3 (前端)

### 1.2 核心功能模块

1. **刷流引擎**: 自动化做种、保种、删种策略
2. **转发引擎**: 多站点间种子转发
3. **辅种引擎** ⭐v3.0核心: 基于三层融合架构的智能辅种系统
4. **站点管理**: 统一的多站点适配器
5. **下载器管理**: qBittorrent / Transmission 支持
6. **通知系统**: Telegram / 邮件 / Webhook

### 1.3 架构决策记录

#### 决策1: 整体架构模式 - 松耦合单体 + 预留插件化

**讨论日期**: 2026-04-12  
**候选方案**:
- A: 标准单体应用 ✅ **选中**
- B: Monorepo + 微服务就绪

**决策理由**:
- ✅ 部署简单：单二进制文件，零依赖
- ✅ 开发效率高：模块间直接函数调用
- ✅ 性能优秀：无网络开销，内存<100MB
- ✅ 资源占用低：适合NAS/VPS环境
- ✅ 运维成本低：systemd即可运行

**应对策略（劣势缓解）**:
- 接口隔离 + 依赖注入（降低耦合）
- 插件化设计（后续可拆分微服务）
- Go goroutine轻松处理1万+并发任务

**参考项目**: ptdog (Go), torrentbotx (Python)

#### 决策2: 数据库选型 - 双数据库策略

**讨论日期**: 2026-04-12  
**最终决策**: SQLite (本地) + MySQL (云端)

**SQLite用途**:
- 本地运行模式的核心数据库
- 存储配置、任务、日志等结构化数据
- 零配置，适合个人使用

**MySQL用途**:
- 云端部署或多人协作场景
- 大规模数据存储和分析
- 高并发读写支持

**切换能力**: 配置即可切换，代码层统一抽象

#### 决策3: MySQL vs PostgreSQL - MySQL 8.0首选

**对比结果**:
| 维度 | MySQL 8.0 | PostgreSQL |
|------|-----------|------------|
| 性能 | ⭐⭐⭐⭐⭐ 胜出30-40% | ⭐⭐⭐⭐ |
| 运维成本 | 低 | 中 |
| PT生态兼容性 | 高（NexusPHP原生支持） | 中 |
| 学习曲线 | 低 | 高 |
| 功能完整性 | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |

**决策理由**: 性能胜出 + 运维简便 + PT生态兼容性好

### 1.4 v2.0 → v3.0 核心升级

| 变化维度 | v2.0 | v3.0 | 来源 |
|----------|------|------|------|
| **匹配架构** | 单一Decision引擎 | **三层融合架构**（精确+模糊+决策） | 全工具综合 |
| **精确匹配** | pieces_hash + IYUU | **pieces_hash批量查询**（ptdog优化） | reseed-puppy-php |
| **模糊匹配** | 无 | **ContentFingerprint多维度匹配** | Graft |
| **文件树对比** | 三种模式 | **±5%容差 + 比例匹配** | Reseed-backend |
| **数据源** | 4种 | **5种+本地指纹数据库** | IYUUPlus + Graft |
| **缓存策略** | 基础缓存 | **三级缓存**（内存+SQLite+云端） | iyuuplus-dev |
| **批量处理** | 200个/批次 | **动态批次+限流控制** | reseed-puppy-php |
| **降级策略** | 无 | **分层降级**（精确→模糊→IYUU） | 全工具综合 |

### 1.5 设计目标

1. **最大化辅种成功率**: 通过多层匹配策略，覆盖更多场景
2. **最小化站点请求**: 本地指纹数据库 + 智能缓存
3. **智能化降级**: 精确匹配失败时自动切换到模糊匹配
4. **用户可控**: 所有关键参数都可配置
5. **隐私保护**: 支持完全本地运行模式

### 2.2 详细策略分析

#### 2.2.1 pieces_hash精确匹配（来自 reseed-puppy-php）

**原理**: 
- pieces_hash 是种子文件中 `info.pieces` 字段的 SHA1 哈希
- 即使不同站点使用不同的 announce URL，只要内容相同，pieces_hash 就相同
- NexusPHP 1.8.5+ 版本提供 pieces_hash 查询接口

**优势**:
- ✅ 准确率极高（99.9%）
- ✅ 完全本地计算，无隐私风险
- ✅ 可批量查询，减少API调用

**劣势**:
- ❌ 需要站点支持 pieces_hash 接口
- ❌ 覆盖率受限于站点支持度

**实现参考**:
```php
// reseed-puppy-php 的实现
$torrent_data = file_get_contents($file_path);
$torrent = Bencode::decode($torrent_data);
$pieces = $torrent['info']['pieces'];
$pieces_sha1 = sha1($pieces);  // pieces_hash
```

**优化点（来自 ptdog）**:
- 批量查询 pieces_hash，减少HTTP请求
- 缓存已查询的 pieces_hash 结果
- 按站点分组查询，避免跨站点限流

#### 2.2.2 IYUU API匹配（来自 iyuuplus-dev）

**原理**:
- 用户上传本地做种的 info_hash 列表到 IYUU 服务器
- IYUU 服务器在其数据库中查找匹配的站点种子
- 返回可辅种的种子信息（站点ID、种子ID、下载链接等）

**优势**:
- ✅ 覆盖率高（60-80%）
- ✅ 无需关心站点API差异
- ✅ 自动维护站点配置

**劣势**:
- ❌ 依赖第三方服务
- ❌ info_hash 上传云端，隐私性差
- ❌ 有API限流（200个/批次）

**实现参考**:
```php
// iyuuplus-dev 的实现
$reseedClient = iyuu_reseed_client();
$result = $reseedClient->reseed($hash_json, $hash_sha1, $sid_sha1, $version);
```

**优化点**:
- 实现TooManyRequestsException处理
- 本地缓存限流时间
- 分批次处理（200个/批次）

#### 2.2.3 ContentFingerprint匹配（来自 Graft）

**原理**:
- 计算种子的内容指纹：
  - 总大小（total_size）
  - 文件数量（file_count）
  - 最大文件大小（largest_file_size）
  - 文件列表哈希（files_hash）
- 通过多维度匹配判断是否为相同内容

**优势**:
- ✅ 完全本地计算，无隐私风险
- ✅ 覆盖率高（70-90%）
- ✅ 不依赖站点API

**劣势**:
- ❌ 准确率相对较低（85%）
- ❌ 需要下载种子元数据
- ❌ 可能产生误匹配

**实现参考**:
```rust
// Graft 的实现
pub struct ContentFingerprint {
    pub total_size: u64,
    pub file_count: usize,
    pub largest_file_size: u64,
    pub files_hash: Option<String>,
}

impl ContentFingerprint {
    pub fn matches(&self, other: &ContentFingerprint) -> MatchResult {
        // 1. 总大小必须精确匹配
        if self.total_size != other.total_size {
            return MatchResult::NoMatch;
        }
        
        // 2. 如果files_hash都存在，精确匹配
        if let (Some(ref hash1), Some(ref hash2)) = (&self.files_hash, &other.files_hash) {
            if hash1 == hash2 {
                return MatchResult::ExactMatch;
            }
        }
        
        // 3. 检查最大文件大小
        if self.largest_file_size != other.largest_file_size {
            return MatchResult::LowConfidence;
        }
        
        // 4. 检查文件数量（允许±2）
        let count_diff = (self.file_count as i64 - other.file_count as i64).abs();
        if count_diff > 2 {
            return MatchResult::LowConfidence;
        }
        
        // 5. 返回匹配结果
        if count_diff == 0 {
            MatchResult::HighConfidence
        } else {
            MatchResult::MediumConfidence
        }
    }
}
```

**置信度分级**:
- **ExactMatch**: files_hash 完全匹配
- **HighConfidence**: 大小+文件数+最大文件都匹配
- **MediumConfidence**: 大小匹配，文件数相差≤2
- **LowConfidence**: 仅大小匹配
- **NoMatch**: 不匹配

#### 2.2.4 文件树对比（来自 Reseed-backend）

**原理**:
- 比较种子文件的路径和大小
- 允许±5%的文件大小容差
- 支持三种匹配模式：
  - STRICT: 路径+大小都匹配
  - FLEXIBLE: 仅大小匹配
  - PARTIAL: 一定比例文件匹配

**优势**:
- ✅ 准确率较高（90%）
- ✅ 可以处理不同命名习惯
- ✅ 支持部分匹配

**劣势**:
- ❌ 需要完整的文件列表
- ❌ 计算量较大
- ❌ 可能误匹配（大小相近的不同内容）

**实现参考**:
```python
# Reseed-backend 的实现
def compare_torrents(name, files):
    torrents = mysql.select_torrent(name)
    cmp_success = []
    cmp_warning = []
    
    for t in torrents:
        torrent_files = eval(t['files'])
        
        if len(torrent_files):
            success_count = failure_count = 0
            
            for filename, expected_size in torrent_files.items():
                actual_size = files.get(filename, -1)
                
                # ±5% 容差比对
                if expected_size * 0.95 < actual_size < expected_size * 1.05:
                    success_count += 1
                else:
                    failure_count += 1
            
            # 判定逻辑
            if failure_count == 0:
                cmp_success.append({'id': t['id']})
            elif success_count > failure_count:
                cmp_warning.append({'id': t['id']})
    
    return {'name': name, 'cmp_success': cmp_success, 'cmp_warning': cmp_warning}
```

**优化点**:
- 使用±5%容差处理文件大小差异
- 支持部分匹配（多数文件匹配即认为匹配）
- 区分完全匹配和部分匹配

#### 2.2.5 info_hash直接匹配

**原理**:
- 通过站点API或爬虫获取站点的种子列表
- 直接比较 info_hash 是否相同
- 如果相同，则可以辅种

**优势**:
- ✅ 准确率100%
- ✅ 实现简单

**劣势**:
- ❌ 覆盖率极低（5-10%）
- ❌ 需要大量API请求
- ❌ 受限流影响严重

**适用场景**:
- 站点提供种子搜索接口
- 特定站点的批量辅种

### 3.2 三层匹配引擎设计

#### 3.2.1 第一层：精确匹配层

**目标**: 快速、准确匹配，覆盖40-60%的场景

**匹配流程**:
```
1. 从下载器获取做种种子列表
2. 解析每个种子，提取 pieces_hash 和 info_hash
3. 并行执行：
   a. pieces_hash 批量查询站点
   b. 调用 IYUU API 匹配
4. 合并结果，去重
5. 返回精确匹配结果
```

**核心接口**:
```go
type ExactMatchEngine interface {
    // pieces_hash 匹配
    MatchByPiecesHash(piecesHashes []string, sites []string) ([]*MatchResult, error)
    
    // IYUU API 匹配
    MatchByIYUU(infoHashes []string) ([]*MatchResult, error)
    
    // 合并结果
    MergeResults(results ...[]*MatchResult) []*MatchResult
}

type MatchResult struct {
    InfoHash       string            `json:"info_hash"`
    PiecesHash     string            `json:"pieces_hash,omitempty"`

```

**优化点**:
- pieces_hash 批量查询：每次最多200个
- IYUU API 限流处理：缓存限流时间
- 并行执行：两个匹配方法同时进行
- 结果去重：相同种子只保留一个最优结果

#### 3.2.2 第二层：模糊匹配层

**目标**: 对精确匹配失败的种子进行模糊匹配，提升覆盖率

**匹配流程**:
```
1. 获取精确匹配失败的种子列表
2. 对每个种子下载元数据（从已知的其他站点）
3. 并行执行：
   a. 计算 ContentFingerprint
   b. 构建文件树
4. 查询本地指纹数据库，找到候选种子
5. 对候选种子执行：
   a. ContentFingerprint 匹配
   b. 文件树对比
6. 过滤低置信度结果（< 0.7）
7. 返回模糊匹配结果
```

**核心接口**:

#### 3.2.3 第三层：决策引擎层

**目标**: 对模糊匹配结果进行最终决策，避免误匹配

**决策流程**:
```
1. 接收模糊匹配结果
2. 对每个结果执行 Decision 评估
3. 过滤掉：
   - BLACKLISTED_RELEASE
   - SAME_INFO_HASH
   - INFO_HASH_ALREADY_EXISTS
   - BLOCKED_RELEASE
4. 对保留的结果进行排序：
   - 按置信度降序
   - 按站点优先级
   - 按种子大小
5. 应用用户配置的规则：
   - 最小置信度阈值
   - 最大辅种数量
   - 站点白名单/黑名单
6. 返回最终决策结果
```

```go
func AssessCandidate(candidate *Candidate, searchee *Searchee, config *MatchConfig) (*Assessment, error) {
    // 1. 黑名单检查
    if isBlacklisted(candidate.Name, config.Blacklist) {
        return &Assessment{Decision: Decision_BLOCKED_RELEASE}, nil
    }
    
    // 2. InfoHash 排重
    if candidate.Metafile.InfoHash == searchee.InfoHash {
        return &Assessment{Decision: Decision_SAME_INFO_HASH}, nil
    }
    if existsInDownloadClient(candidate.Metafile.InfoHash) {
        return &Assessment{Decision: Decision_INFO_HASH_ALREADY_EXISTS}, nil
    }
    
    // 3. 大小匹配（±容差）
    if !fuzzySizeMatches(candidate.Metafile.TotalSize, searchee.TotalSize, config.FuzzySizeTolerance) {
        return &Assessment{Decision: Decision_FUZZY_SIZE_MISMATCH}, nil
    }
    
    // 4. 文件树对比
    fileTreeResult := CompareFileTrees(candidate.Metafile.Files, searchee.Files)
    if !fileTreeResult.Matches(config.FileTreeMode) {
        return &Assessment{Decision: Decision_FILE_TREE_MISMATCH}, nil
    }
    
    // 5. 返回匹配结果
    decision := Decision_MATCH
    if !fileTreeResult.PerfectMatch {
        decision = Decision_MATCH_PARTIAL
    }
    
    return &Assessment{
        Decision:      decision,
        Metafile:      candidate.Metafile,
        InfoHashMatch: false,
        Confidence:    fileTreeResult.Confidence,
    }, nil
}
```

### 3.3 智能降级策略

**降级链**:
```
Level 1: pieces_hash 精确匹配（覆盖 40-60%）
    ↓ 失败
Level 2: IYUU API 匹配（额外覆盖 20-30%）
    ↓ 失败
Level 3: ContentFingerprint 模糊匹配（额外覆盖 10-20%）
    ↓ 失败
Level 4: 文件树对比（额外覆盖 5-10%）
    ↓ 失败
Level 5: 记录失败原因，跳过该种子
```

```yaml
        enabled: true
```

### 5.3 新增表详解（v3.0核心）

#### 5.3.1 content_fingerprints 表（内容指纹数据库）⭐核心

**用途**: 存储所有已知种子的内容指纹，用于模糊匹配

**来源**: Graft 的本地索引设计

```sql
CREATE TABLE content_fingerprints (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    total_size BIGINT NOT NULL,
    file_count INTEGER NOT NULL,
    largest_file_size BIGINT NOT NULL,
    files_hash VARCHAR(40),
    info_hash VARCHAR(40) NOT NULL UNIQUE,
    torrent_name VARCHAR(500),
    site_id INTEGER,
    torrent_id INTEGER,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    
    INDEX idx_total_size (total_size),
    INDEX idx_file_count (file_count),
    INDEX idx_files_hash (files_hash),
    INDEX idx_info_hash (info_hash),
    INDEX idx_site_torrent (site_id, torrent_id)
);

CREATE INDEX idx_fingerprint_composite ON content_fingerprints(total_size, file_count, largest_file_size);
```

**字段说明**:
- `total_size`: 种子总大小（字节）
- `file_count`: 文件数量
- `largest_file_size`: 最大文件大小（字节）
- `files_hash`: 文件列表的 SHA1 哈希
- `info_hash`: 种子的 info_hash
- `torrent_name`: 种子名称
- `site_id`: 站点ID
- `torrent_id`: 种子ID

#### 5.3.2 reseed_matches 表（匹配结果缓存）⭐核心

**用途**: 缓存匹配结果，避免重复匹配


### 5.5 索引优化策略

#### 5.5.1 复合索引设计

```sql
-- content_fingerprints 表：模糊匹配查询优化
CREATE INDEX idx_fingerprint_search ON content_fingerprints(total_size, file_count, largest_file_size);
-- 查询模式: WHERE total_size = ? AND file_count BETWEEN ?-2 AND ?+2 AND largest_file_size = ?

-- reseed_matches 表：待注入结果查询
CREATE INDEX idx_match_composite ON reseed_matches(source_info_hash, status, confidence DESC);
-- 查询模式: WHERE source_info_hash = ? AND status = 'pending' ORDER BY confidence DESC

-- torrent_hashes 表：站点查询优化
CREATE INDEX idx_hash_composite ON torrent_hashes(pieces_hash, site_id);
-- 查询模式: WHERE pieces_hash = ? AND site_id = ?
```

#### 5.5.2 分区表设计（MySQL大规模部署）

```sql
-- reseed_matches 按月分区（数据量>100万时启用）
CREATE TABLE reseed_matches_partitioned (
    id BIGINT AUTO_INCREMENT,
    execution_id BIGINT,
    source_info_hash VARCHAR(40) NOT NULL,
    -- ... 其他字段 ...

    INDEX idx_source_hash (source_info_hash),
    INDEX idx_execution_id (execution_id),
    INDEX idx_status (status)
) PARTITION BY RANGE (YEAR(created_at) * 100 + MONTH(created_at)) (
    PARTITION p202604 VALUES LESS THAN (202605),
    PARTITION p202605 VALUES LESS THAN (202606),
    PARTITION p202606 VALUES LESS THAN (202607),
    PARTITION p_future VALUES LESS THAN MAXVALUE
);
```

```go
// 三级缓存
type CacheManager struct {
    l1Cache *cache.Cache  // 内存缓存（LRU）
    l2DB    *sql.DB       // SQLite 缓存
    l3DB    *sql.DB       // MySQL 缓存
}

func (c *CacheManager) Get(key string) (interface{}, bool) {
    // L1: 内存缓存
    if val, found := c.l1Cache.Get(key); found {
        return val, true
    }
    
    // L2: SQLite 缓存
    if val, found := c.getFromL2(key); found {
        c.l1Cache.Set(key, val, cache.DefaultExpiration)
        return val, true
    }
    
    // L3: MySQL 缓存
    if val, found := c.getFromL3(key); found {
        c.setL2(key, val)
        c.l1Cache.Set(key, val, cache.DefaultExpiration)
        return val, true
    }
    
    return nil, false
}
```

```go
type HTTPClient struct {

func NewHTTPClient() *HTTPClient {
    return &HTTPClient{
        client: &http.Client{
            Transport: &http.Transport{
                MaxIdleConns:        100,

        },

// 批量请求
func BatchRequest(urls []string) ([]*http.Response, error) {
    var wg sync.WaitGroup
    results := make(chan *http.Response, len(urls))
    errors := make(chan error, len(urls))
    
    semaphore := make(chan struct{}, 10)  // 限制并发数
    
    for _, url := range urls {
        wg.Add(1)
        go func(u string) {
            defer wg.Done()
            semaphore <- struct{}{}
            defer func() { <-semaphore }()
            
            resp, err := http.Get(u)
            if err != nil {
                errors <- err
                return
            }
            results <- resp
        }(url)
    }
    
    wg.Wait()
    close(results)
    close(errors)
    
    var responses []*http.Response
    for resp := range results {
        responses = append(responses, resp)
    }
    
    if len(errors) > 0 {
        return responses, <-errors
    }
    
    return responses, nil
}
```

## 十、设计决策与讨论记录

### 10.1 核心设计决策

#### 决策1: 采用三层融合架构

**问题**: 如何最大化辅种覆盖率同时保证准确率？

**讨论过程**:
- 单一匹配方法无法满足所有场景
- pieces_hash准确率高但覆盖率低（40-60%）
- IYUU覆盖率高但依赖第三方服务（60-80%）
- 需要多种方法结合，从精确到模糊逐级匹配

**决策结果**: 采用三层融合架构
```
Level 1: 精确匹配层（pieces_hash + IYUU）
    ↓ 失败
Level 2: 模糊匹配层（ContentFingerprint + 文件树）
    ↓ 失败
Level 3: 决策引擎层（15种Decision）
```

**理由**:
- 清晰的职责分离，易于维护和扩展
- 可以独立优化每层性能
- 灵活的降级策略，避免资源浪费

#### 决策2: 引入智能降级策略

**问题**: 如何平衡性能和覆盖率？

**讨论过程**:
- 所有方法都执行会很慢（1000种子需180秒）
- 需要根据场景选择合适的匹配方法
- 需要避免重复的API调用和计算

**决策结果**: 实现可配置的智能降级策略
```yaml
fallback:
  enabled: true
  levels:
    - method: pieces_hash
      enabled: true
      confidence_threshold: 1.0
    - method: iyuu
      enabled: true
      confidence_threshold: 0.95
    - method: fingerprint
      enabled: true
      confidence_threshold: 0.7
    - method: file_tree
      enabled: true
      confidence_threshold: 0.75
  max_fallbacks: 3
```

**理由**:
- 精确匹配失败才降级到模糊匹配
- 用户可配置降级次数和阈值
- 支持首次匹配即停止的模式

#### 决策3: 构建本地指纹数据库

**问题**: 如何保护用户隐私并减少站点API调用？

**讨论过程**:
- IYUU需要上传info_hash到云端，隐私性差
- ContentFingerprint需要快速查询支持
- 离线运行需要本地数据源

**决策结果**: 实现content_fingerprints表，构建本地索引
```sql
CREATE TABLE content_fingerprints (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    total_size BIGINT NOT NULL,
    file_count INTEGER NOT NULL,
    largest_file_size BIGINT NOT NULL,
    files_hash VARCHAR(40),
    info_hash VARCHAR(40) NOT NULL UNIQUE,
    torrent_name VARCHAR(500),
    site_id INTEGER,
    torrent_id INTEGER,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

**理由**:
- 支持完全本地运行，保护隐私
- 减少站点API调用，提升性能
- 离线模式的基础设施

#### 决策4: 实现三级缓存机制

**问题**: 如何减少重复计算和API调用？

**讨论过程**:
- pieces_hash查询结果可以复用
- IYUU API有限流，需要避免重复请求
- 匹配结果可以缓存，避免重复匹配

**决策结果**: 实现三级缓存架构
```
Level 1: 内存缓存（LRU，最快）
    └─ 缓存：匹配结果、pieces_hash查询结果
Level 2: SQLite缓存（持久化，中等）
    └─ 缓存：reseed_cache、iyuu_rate_limit
Level 3: MySQL缓存（云端，最慢）
    └─ 缓存：跨设备共享的匹配结果
```

**理由**:
- 减少重复计算，提升响应速度
- 减少API调用，避免限流
- 支持跨设备共享缓存

**Go 实现示例**:
```go
type FileHashInput struct {
    Path string
    Name string
    Size int64
}

func CalculateFilesHash(files []FileHashInput) string {
    if len(files) == 0 {
        return ""
    }

    // 1. 排序
    sorted := make([]FileHashInput, len(files))
    copy(sorted, files)
    
    sort.Slice(sorted, func(i, j int) bool {
        if sorted[i].Path != "" && sorted[j].Path != "" {
            if sorted[i].Path != sorted[j].Path {
                return sorted[i].Path < sorted[j].Path
            }
        }
        return sorted[i].Name < sorted[j].Name
    })

    // 2. 计算 SHA1
    h := sha1.New()
    for _, file := range sorted {
        if file.Path != "" {
            h.Write([]byte(file.Path))
        }
        h.Write([]byte(file.Name))
        buf := make([]byte, 8)
        binary.LittleEndian.PutUint64(buf, uint64(file.Size))
        h.Write(buf)
    }

    return hex.EncodeToString(h.Sum(nil))
}
```

**决策理由**:
- ✅ Graft 的排序方式保证了跨平台一致性
- ✅ cross-seed 的路径处理更灵活，适应不同场景
- ✅ 小端序与 Graft 保持一致，便于后续集成
- ✅ SHA1 算法性能好，碰撞概率低

**重要原则**: **PT不允许文件有差异，SHA1必须一致才能认为是同一个文件或资源！**

**误区澄清**:

**种子元数据可能差异的来源**：
- 不同的 announce URL
- 不同的 creation date

- 不同的 publisher 信息

**最终决策**:

| 匹配维度 | 要求 | 容差 | 理由 |
|---------|------|------|------|
| **文件大小** | **完全一致** | **0%** | PT核心原则 |
| **文件SHA1** | **完全一致** | **0%** | 内容验证 |
| **文件数量** | **完全一致** | **0%** | 结构验证 |
| **种子总大小** | **允许差异** | **±1%** | 仅限元数据差异 |

**分层匹配策略**：

```
Layer 1: 精确匹配（最高优先级）
├── pieces_hash 匹配 → 确认内容完全一致 ✅
└── info_hash 匹配 → 确认种子完全一致 ✅

Layer 2: 元数据匹配（次优先级）
├── 种子总大小 ±1% → 允许元数据差异
├── 文件树结构 → 必须完全一致
└── files_hash → 必须完全一致

Layer 3: 决策过滤
├── 排除元数据差异过大的种子
├── 排除文件结构不同的种子
└── 最终确认可辅种
```

```go
// 单个条目匹配
func matchesBlocklist(torrent *Torrent, entry BlocklistEntry) bool {
    switch entry.Type {
    case BlocklistTypeName:
        return strings.Contains(torrent.Name, entry.Value)
        
    case BlocklistTypeNameRegex:
        matched, _ := regexp.MatchString(entry.Value, torrent.Name)
        return matched
        
    case BlocklistTypeFolder:
        if torrent.Path == "" {
            return false
        }
        dir := filepath.Dir(torrent.Path)
        return strings.Contains(dir, entry.Value)
        
    case BlocklistTypeFolderRegex:
        if torrent.Path == "" {
            return false
        }
        dir := filepath.Dir(torrent.Path)
        matched, _ := regexp.MatchString(entry.Value, dir)
        return matched
        
    case BlocklistTypeCategory:
        return torrent.Category == entry.Value
        
    case BlocklistTypeTag:
        for _, tag := range torrent.Tags {
            if tag == entry.Value {
                return true
            }
        }
        return false
        
    case BlocklistTypeTracker:
        for _, tracker := range torrent.Trackers {
            // 提取主机名
            u, err := url.Parse(tracker)
            if err == nil && u.Hostname() == entry.Value {
                return true
            }
        }
        return false
        
    case BlocklistTypeInfoHash:
        return torrent.InfoHash == entry.Value
        
    case BlocklistTypeSizeBelow:
        size, _ := strconv.ParseInt(entry.Value, 10, 64)
        return torrent.TotalSize < size
        
    case BlocklistTypeSizeAbove:
        size, _ := strconv.ParseInt(entry.Value, 10, 64)
        return torrent.TotalSize > size
        
    case BlocklistTypeLegacy:
        // 兼容模式：匹配名称或info_hash或文件夹
        if strings.Contains(torrent.Name, entry.Value) {
            return true
        }
        if torrent.InfoHash == entry.Value {
            return true
        }
        if torrent.Path != "" {
            dir := filepath.Dir(torrent.Path)
            if strings.Contains(dir, entry.Value) {
                return true
            }
        }
        return false
        
    default:
        return false
    }
}
```

```go
func QueryFingerprintOptimized(fp ContentFingerprint) ([]MatchResult, error) {
    // Step 1: 检查内存缓存
    if cached, ok := GetCachedQuery(fp); ok {
        return cached, nil
    }
    
    // Step 2: 布隆过滤器快速排除
    if fp.PiecesHash != "" {
        if !MayExistByPiecesHash(fp.PiecesHash) {
            return []MatchResult{}, nil // 肯定不存在
        }
    }
    
    // Step 3: Redis 缓存
    cacheKey := fmt.Sprintf("fp:%s", fp.PiecesHash)
    if cached, err := redisClient.Get(ctx, cacheKey).Result(); err == nil {
        var results []MatchResult
        json.Unmarshal([]byte(cached), &results)
        return results, nil
    }
    
    // Step 4: 数据库查询（分层查询）
    var results []MatchResult
    
    // Layer 1: pieces_hash 精确匹配
    if fp.PiecesHash != "" {
        exactMatches := QueryByPiecesHash(fp.PiecesHash)
        if len(exactMatches) > 0 {
            results = append(results, exactMatches...)
            goto CacheAndReturn
        }
    }
    
    // Layer 2: files_hash 精确匹配
    if fp.FilesHash != "" {
        if MayExistByFilesHash(fp.FilesHash) {
            fileMatches := QueryByFilesHash(fp.FilesHash)
            results = append(results, fileMatches...)
        }
    }
    
    // Layer 3: 复合条件模糊匹配
    fuzzyMatches := QueryByFingerprint(fp)
    results = append(results, fuzzyMatches...)
    
CacheAndReturn:
    // Step 5: 缓存结果
    CacheQuery(fp, results)
    CacheQueryResult(fp.QueryHash(), results)
    
    return results, nil
}
```

#### 2.1 信号量实现

```go
import "golang.org/x/sync/semaphore"
import "context"

// 并发控制器
type ConcurrencyController struct {
    config  *ConcurrencyConfig
    
    // 全局信号量
    globalSemaphore *semaphore.Weighted
    
    // 站点信号量
    siteSemaphores map[string]*semaphore.Weighted
    siteMutex      sync.RWMutex
    
    // 请求限流器
    rateLimiters map[string]*rate.Limiter
    rateMutex    sync.RWMutex
    
    // 动态调整
    currentConcurrent int
    performanceMetrics *PerformanceMetrics
}

func NewConcurrencyController(config *ConcurrencyConfig) *ConcurrencyController {
    ctrl := &ConcurrencyController{
        config:            config,
        globalSemaphore:   semaphore.NewWeighted(int64(config.GlobalMaxConcurrent)),
        siteSemaphores:    make(map[string]*semaphore.Weighted),
        rateLimiters:      make(map[string]*rate.Limiter),
        currentConcurrent: 0,
        performanceMetrics: NewPerformanceMetrics(),
    }
    
    return ctrl
}

// 获取站点信号量
func (c *ConcurrencyController) GetSiteSemaphore(siteID string) *semaphore.Weighted {
    c.siteMutex.RLock()
    sem, ok := c.siteSemaphores[siteID]
    c.siteMutex.RUnlock()
    
    if !ok {
        c.siteMutex.Lock()
        // 双重检查
        sem, ok = c.siteSemaphores[siteID]
        if !ok {
            // 使用站点特定配置或默认值
            concurrent := c.config.TorrentConcurrentPerSite
            if siteConcurrent, ok := c.config.SiteConcurrent[siteID]; ok {
                concurrent = siteConcurrent
            }
            sem = semaphore.NewWeighted(int64(concurrent))
            c.siteSemaphores[siteID] = sem
        }
        c.siteMutex.Unlock()
    }
    
    return sem
}

// 获取站点限流器
func (c *ConcurrencyController) GetRateLimiter(siteID string) *rate.Limiter {
    c.rateMutex.RLock()
    limiter, ok := c.rateLimiters[siteID]
    c.rateMutex.RUnlock()
    
    if !ok {
        c.rateMutex.Lock()
        limiter, ok = c.rateLimiters[siteID]
        if !ok {
            // 使用站点特定配置或默认值
            delay := c.config.DefaultRequestDelay
            if siteDelay, ok := c.config.RequestDelayPerSite[siteID]; ok {
                delay = siteDelay
            }
            // 计算每秒请求数
            rps := float64(time.Second) / float64(delay)
            limiter = rate.NewLimiter(rate.Limit(rps), 1)
            c.rateLimiters[siteID] = limiter
        }
        c.rateMutex.Unlock()
    }
    
    return limiter
}

// 执行带并发控制的任务
func (c *ConcurrencyController) Execute(
    ctx context.Context,
    siteID string,
    task func() error,
) error {
    // 1. 获取全局信号量
    if err := c.globalSemaphore.Acquire(ctx, 1); err != nil {
        return err
    }
    defer c.globalSemaphore.Release(1)
    
    // 2. 获取站点信号量
    siteSem := c.GetSiteSemaphore(siteID)
    if err := siteSem.Acquire(ctx, 1); err != nil {
        return err
    }
    defer siteSem.Release(1)
    
    // 3. 请求限流
    limiter := c.GetRateLimiter(siteID)
    if err := limiter.Wait(ctx); err != nil {
        return err
    }
    
    // 4. 执行任务
    atomic.AddInt32(&c.currentConcurrent, 1)
    startTime := time.Now()
    
    defer func() {
        atomic.AddInt32(&c.currentConcurrent, -1)
        duration := time.Since(startTime)
        c.performanceMetrics.RecordTask(duration, siteID)
    }()
    
    return task()
}
```

```go
// 批量处理种子
func (c *ConcurrencyController) ProcessTorrentsBatch(
    ctx context.Context,
    torrents []Torrent,
    siteID string,
    processor func(Torrent) error,
) error {
    // 分批处理
    for i := 0; i < len(torrents); i += c.config.BatchSize {
        batch := torrents[i:min(i+c.config.BatchSize, len(torrents))]
        
        // 并发处理批次内的种子
        var wg sync.WaitGroup
        errChan := make(chan error, len(batch))
        
        for _, torrent := range batch {
            wg.Add(1)
            go func(t Torrent) {
                defer wg.Done()
                
                err := c.Execute(ctx, siteID, func() error {
                    return processor(t)
                })
                
                if err != nil {
                    errChan <- err
                }
            }(torrent)
        }
        
        wg.Wait()
        close(errChan)
        
        // 检查错误
        for err := range errChan {
            if err != nil {
                return err
            }
        }
        
        // 批次间延迟
        if i+c.config.BatchSize < len(torrents) {
            time.Sleep(100 * time.Millisecond)
        }
    }
    
    return nil
}
```

```go
        results, err := c.MatchByIYUU(ctx, torrent, sites)
        if err != nil {
            errChan <- err
        } else {
            resultChan <- results
        }
    }()
    
    go func() {
        defer wg.Done()
        results, err := c.MatchByContentFingerprint(ctx, torrent, sites)
        if err != nil {
            errChan <- err
        } else {
            resultChan <- results
        }

        if err != nil {
            errChan <- err
        } else {
            resultChan <- results
        }
    }()
    
    go func() {
        defer wg.Done()
        results, err := c.MatchByInfoHash(ctx, torrent, sites)
        if err != nil {
            errChan <- err
        } else {
            resultChan <- results
        }
    }()
    
    // 等待所有方法完成

        wg.Wait()
        close(resultChan)
        close(errChan)

    var allResults []MatchResult
    var errors []error
    
    for results := range resultChan {

        }
```

#### 3.1 性能监控

```go
type PerformanceMetrics struct {
    mu sync.RWMutex
    
    // 任务统计
    TotalTasks      int64
    SuccessTasks    int64
    FailedTasks     int64
    
    // 时间统计
    AvgTaskDuration time.Duration
    MaxTaskDuration time.Duration
    MinTaskDuration time.Duration
    
    // 站点统计
    SiteStats map[string]*SiteStats
}

type SiteStats struct {
    TotalTasks      int64
    AvgDuration     time.Duration
    ErrorRate       float64
    LastErrorTime   time.Time
}

func NewPerformanceMetrics() *PerformanceMetrics {
    return &PerformanceMetrics{
        SiteStats: make(map[string]*SiteStats),
    }
}

func (m *PerformanceMetrics) RecordTask(duration time.Duration, siteID string) {
    m.mu.Lock()
    defer m.mu.Unlock()
    
    // 更新全局统计
    atomic.AddInt64(&m.TotalTasks, 1)
    
    if duration > m.MaxTaskDuration {
        m.MaxTaskDuration = duration
    }
    if m.MinTaskDuration == 0 || duration < m.MinTaskDuration {
        m.MinTaskDuration = duration
    }
    
    // 计算平均时间
    totalDuration := m.AvgTaskDuration * time.Duration(atomic.LoadInt64(&m.TotalTasks)-1)
    m.AvgTaskDuration = (totalDuration + duration) / time.Duration(atomic.LoadInt64(&m.TotalTasks))
    
    // 更新站点统计
    if m.SiteStats[siteID] == nil {
        m.SiteStats[siteID] = &SiteStats{}
    }
    
    stats := m.SiteStats[siteID]
    stats.TotalTasks++
    stats.AvgDuration = (stats.AvgDuration*time.Duration(stats.TotalTasks-1) + duration) / time.Duration(stats.TotalTasks)
}

func (m *PerformanceMetrics) RecordError(siteID string) {
    m.mu.Lock()
    defer m.mu.Unlock()
    
    atomic.AddInt64(&m.FailedTasks, 1)
    
    if stats, ok := m.SiteStats[siteID]; ok {
        stats.LastErrorTime = time.Now()
        stats.ErrorRate = float64(m.FailedTasks) / float64(atomic.LoadInt64(&m.TotalTasks))
    }
}
```

#### 3.2 动态调整算法

```go
// 动态调整并发数
func (c *ConcurrencyController) DynamicAdjust() {
    if !c.config.EnableDynamicAdjustment {
        return
    }
    
    metrics := c.performanceMetrics
    metrics.mu.RLock()
    avgDuration := metrics.AvgTaskDuration
    errorRate := float64(atomic.LoadInt64(&metrics.FailedTasks)) / float64(atomic.LoadInt64(&metrics.TotalTasks))
    metrics.mu.RUnlock()
    
    current := atomic.LoadInt32(&c.currentConcurrent)
    
    // 调整策略
    var newConcurrent int
    
    if errorRate > 0.1 {
        // 错误率过高，降低并发
        newConcurrent = int(float64(current) * 0.8)
    } else if avgDuration > 5*time.Second {
        // 任务耗时过长，降低并发
        newConcurrent = int(float64(current) * 0.9)
    } else if avgDuration < 1*time.Second && errorRate < 0.01 {
        // 性能良好，提高并发
        newConcurrent = int(float64(current) * 1.1)
    } else {
        // 保持当前并发
        newConcurrent = int(current)
    }
    
    // 限制范围
    if newConcurrent < c.config.MinConcurrent {
        newConcurrent = c.config.MinConcurrent
    }
    if newConcurrent > c.config.MaxConcurrent {
        newConcurrent = c.config.MaxConcurrent
    }
    
    // 更新全局信号量
    if newConcurrent != int(current) {
        c.globalSemaphore = semaphore.NewWeighted(int64(newConcurrent))
        log.Printf("Adjusted concurrency from %d to %d", current, newConcurrent)
    }
}

// 定期执行动态调整
func (c *ConcurrencyController) StartDynamicAdjustment(interval time.Duration) {
    ticker := time.NewTicker(interval)
    go func() {
        for range ticker.C {
            c.DynamicAdjust()
        }
    }()
}
```

```go
        // 主排序
        cmp := compareByField(results[i], results[j], config.PrimaryField)
        if cmp != 0 {
            if config.PrimaryOrder == "asc" {

            }
            return cmp > 0
        }

        cmp = compareByField(results[i], results[j], config.SecondaryField)
        if config.SecondaryOrder == "asc" {
            return cmp < 0
        }
        return cmp > 0
    })
    
    return results
}
```

```go
    UserPreferences []string `json:"user_preferences"` // 用户偏好
    RecentHistory  []string `json:"recent_history"`   // 最近操作
    SuccessRate    map[string]float64 `json:"success_rate"` // 各站点成功率

// 计算推荐分数
func CalculateRecommendationScore(result MatchResult, config *RecommendationConfig) float64 {
    score := 0.0
    
    // 1. 置信度权重
    highRatio := float64(result.HighConfidence) / float64(result.TotalMatches)
    score += highRatio * 0.4
    
    // 2. 匹配数量权重
    matchScore := math.Min(float64(result.TotalMatches)/10.0, 1.0)
    score += matchScore * 0.2
    
    // 3. 站点成功率权重
    siteScore := 0.0
    for _, site := range result.MatchedSites {
        if rate, ok := config.SuccessRate[site]; ok {
            siteScore += rate
        }
    }
    if len(result.MatchedSites) > 0 {
        siteScore /= float64(len(result.MatchedSites))
    }
    score += siteScore * 0.2
    
    // 4. 时间权重（最近的优先）
    daysSinceMatch := time.Since(result.MatchedAt).Hours() / 24
    timeScore := math.Exp(-daysSinceMatch / 7.0) // 7天衰减
    score += timeScore * 0.2
    
    return score
}

// 推荐排序
func SortByRecommendation(results []MatchResult, config *RecommendationConfig) []MatchResult {
    // 计算每个结果的推荐分数
    scoredResults := make([]struct {
        result MatchResult
        score  float64
    }, len(results))
    
    for i, result := range results {
        scoredResults[i].result = result
        scoredResults[i].score = CalculateRecommendationScore(result, config)
    }
    
    // 按推荐分数排序
    sort.Slice(scoredResults, func(i, j int) bool {
        return scoredResults[i].score > scoredResults[j].score
    })
    
    // 提取排序后的结果
    sortedResults := make([]MatchResult, len(results))
    for i, sr := range scoredResults {
        sortedResults[i] = sr.result
    }
    
    return sortedResults
}
```

### ADR-01 数据库选型：SQLite + MySQL 双数据库架构

**决策日期**: 2026-04-12

**讨论背景**: 辅种功能涉及海量 Info_Hash 存储（可能达千万级），不同匹配方案对数据库需求差异极大。

**核心分析 — 四种匹配方案的数据库需求差异**:

| 匹配方案 | 数据量级 | 查询模式 | SQLite 可行性 | MySQL 必要性 |
|----------|----------|----------|--------------|-------------|
| Info_Hash 精确匹配 | 10万-1000万条 | `WHERE info_hash = 'abc'` | ✅ 百万级 UNIQUE INDEX <1ms | 可选 |
| Pieces_Hash 批量查询 ⭐ | 100万-1000万条 | `WHERE pieces_hash IN (100个)` | ❌ 150-300ms, <5 QPS | ✅ 8-20ms, 400+ QPS |
| 内容指纹识别 | 50万-500万条 | 多条件组合查询 | ✅ <100万可行 | ✅ 大规模更优 |
| 文件树对比 | 10万-100万条 | 应用层计算 | ✅ 完全胜任 | 不需要 |

**结论**: Pieces_Hash 批量查询是核心瓶颈，必须用 MySQL 级别数据库。

**最终决策**:
- **客户端本地**: SQLite（零配置、隐私保护、离线可用）
- **云端服务**: MySQL 8.0 InnoDB（高并发、读写分离、海量存储）
- **统一抽象**: GORM + Repository Pattern，未来可平滑迁移到 PostgreSQL

### ADR-02 MySQL vs PostgreSQL 选型对比

**决策日期**: 2026-04-12

**讨论背景**: 已确定双数据库架构，需在 MySQL 和 PostgreSQL 之间做最终选择。

**核心对比（针对 PT-Forward 场景）**:

| 评估维度 | 权重 | MySQL 8.0 | PostgreSQL 16 | 说明 |
|----------|------|-----------|---------------|------|
| 查询性能 | 25% | 9/10 | 8/10 | InfoHash 查询快 33%，批量 IN 快 43% |
| 写入性能 | 15% | 9/10 | 8/10 | 批量插入快 20%，12000 vs 9500 inserts/s |
| 高并发能力 | 15% | 9/10 | 8.5/10 | 更稳定的高并发表现 |
| 运维简易性 | 15% | 9/10 | 7/10 | Docker 部署更简单，监控工具更成熟 |
| Go 生态支持 | 10% | 9/10 | 9/10 | GORM 让两者体验几乎相同 |
| 功能丰富度 | 10% | 7/10 | 10/10 | PG 的全文搜索/GIS 当前不需要 |
| **加权总分** | **100%** | **8.70** | **8.325** | **MySQL 胜出** |

**关键发现**:
- PT-Forward 当前不需要 PostgreSQL 的独特强项（全文搜索、GIS、复杂 JSON 查询）
- MySQL 在核心场景（批量 IN 查询、高并发写入）性能优势明显
- GORM 抽象层使切换成本极低，保留了未来灵活性

**最终决策**: MySQL 8.0 首选，保留 PostgreSQL 切换能力。何时考虑切换：需要全文搜索、复杂数据分析、GIS 功能时。

### ADR-03 辅种引擎 Pipeline 架构设计

**决策日期**: 2026-04-13

**讨论背景**: 辅种引擎是系统核心差异化功能，需设计完整的 Pipeline 流水线、数据源优先级和匹配算法链。

**五阶段 Pipeline 架构**:

| 阶段 | 名称 | 核心职责 | 耗时(500种子) |
|------|------|----------|--------------|
| Stage 1 | Collector | 并发从所有下载器收集种子列表 | ~3-5s |
| Stage 2 | Extractor | 为每个 Searchee 计算 InfoHash/PiecesHash/文件树/指纹 | ~10-30s |
| Stage 3 | DatasourceScheduler | 并发查询所有数据源，合并去重 | ~2-5s |
| Stage 4 | DecisionEngine | 15 种 Decision 结果精准评估 | ~5-15s |
| Stage 5 | Executor | 速率限制下的安全注入 | ~30-60s |

**数据源优先级链（5 层）**:
```
L0: Memory Cache (<0.1ms)  → 最近 10000 条查询结果
L1: Local SQLite (1-5ms)   → 本地缓存种子元数据
L2: Cloud MySQL (10-50ms)  → 社区共享海量 hash 库 ⭐ 核心
L3: IYUU API (200-1000ms)  → 第三方补充数据源
L4: Site API (500-2000ms)  → 站点 pieces-hash 接口
```

**降级策略**: CloudMySQL 失败 → 自动降级到 LocalSQLite + IYUU → 所有远程失败 → 仅本地数据。

**四级匹配算法链**: InfoHash(O(1)) → PiecesHash(O(n)) → Fingerprint(O(n log n)) → FileTree(O(n*m))

**v3.0 演进说明**: v3.0 在此基础上进化为"三层融合辅种引擎"（精确层 + 模糊层 + 决策引擎层），但 Pipeline 五阶段架构和降级策略的核心设计保持不变。

### ADR-04 API 设计规范

**决策日期**: 2026-04-13

**讨论背景**: 需要兼顾开发效率、可维护性和灵活扩展能力的 API 架构。

**三种方案对比**:

| 方案 | 灵活性 | 复杂度 | 适用性 |
|------|--------|--------|--------|
| A: 传统静态路由 | ❌ 低 | 低 | 简单项目 |
| B: 增强型模块化路由 ⭐ | ✅ 高 | 中 | **PT-Forward** |
| C: GraphQL | ✅ 高 | 高 | 过度工程 |

**最终决策**: RESTful Level 2 + 增强型模块化路由架构

**核心设计**:
1. **Module 接口抽象**: `Name()/BasePath()/RegisterRoutes()/Middlewares()` — 新模块只需实现接口并注册
2. **ModuleRegistry 注册表**: 运行时查询所有路由、启用/禁用模块
3. **YAML 配置驱动**: 通过配置文件控制模块开关，无需重编译
4. **统一响应格式**: `{code, message, data, meta, error}` 双重编码（HTTP Status + Business Code）
5. **业务码体系**: 通用(1xxxx)、站点(51xxx)、下载器(52xxx)、辅种(53xxx)、刷流(54xxx)、服务器(5xxxx)

**渐进式演进路径**:
```
Phase 1: 内置 Module + Registry + YAML 配置
Phase 2: .so 插件加载 + Plugin Router
Phase 3: 分布式模块注册 + 服务发现
```

### ADR-05 并发模型与容错机制

**决策日期**: 2026-04-13

**讨论背景**: PT 站有严格反作弊系统，异常行为导致封号；系统需 7×24 小时稳定运行。

**三种方案对比**:

| 方案 | 安全性 | 可控性 | 说明 |
|------|--------|--------|------|
| A: 原生 Goroutine | ❌ | ❌ | 无并发控制、OOM 风险、无法限速 |
| B: 基础 Worker Pool | ⚠️ | ⚠️ | 缺少速率限制、断路器、降级能力 |
| C: 分层协程池 ⭐ | ✅ | ✅ | 场景化优化 + 三层容错 + 可配置速率 |

**最终决策**: 分层协程池架构 + 三层容错 + 用户可配置注入速率

**7 个专用协程池**:

| 池名称 | Worker 数 | QPS 限制 | 用途 |
|--------|----------|----------|------|
| extractor | CPU×2 | 无限 | 特征提取(CPU 密集) |
| datasource-query | 20 | 10 | 多源查询(I/O 密集) |
| decision-engine | CPU×4 | 无限 | Decision 匹配(CPU 密集) |
| **injector** | **5** | **可配置** | **种子注入(速率敏感)** ⭐ |
| site-sync | 15 | 5 | 站点同步 |
| notification | 5 | 无限 | 通知推送 |
| hash-compute | CPU | 无限 | Hash 计算 |

**三层容错机制**:
1. **应用层重试**: 指数退避，max 3 次，针对瞬态故障(502/503/timeout)
2. **协程池层重试**: 透明重试，max 2-3 次，对上层无感
3. **断路器模式**: 连续 5 次失败后熔断，30s 后半开探测，防止级联雪崩

**种子注入速率控制（核心安全特性）**:
- 默认: **15 秒/个 ± 5 秒随机抖动**（模拟人类行为）
- 用户可通过 UI/API 动态调整: base_interval(5-300s)、jitter_range(0-60s)
- 三种预设模式: Safe(30s±15s) / Normal(15s±5s) / Aggressive(5s±3s)
- 支持每站点独立覆盖（如 M-Team 20s±10s, Gazelle-HD 30s±15s）

**自动降级**: Normal(全功能) → Partial(高优先级) → Minimal(仅核心) → Emergency(只读)，基于 CPU/内存/Goroutine 数自动触发。
