# §34 辅种引擎 L2 搜索验证匹配设计方案

> 状态：设计中
> 日期：2026-06-10
> 关联：§33 辅种引擎、§30.5 全局转载规则

---

## 1. 背景与问题

### 1.1 现状

辅种引擎当前有 4 层匹配：

| Layer | 方法 | 依赖 | 对无 pieces_hash API 站点可用？ |
|-------|------|------|-------------------------------|
| L0 | pieces_hash | 站点 API | **否** |
| L1.5 | file_tree | 本地指纹 DB | **否**（需预先采集目标站 .torrent） |
| L2 | size_title | 站点搜索接口 | **部分**（只做标题+大小模糊匹配，精度低） |
| L3 | fingerprint | 本地指纹 DB | **否**（同 L1.5） |

验证结果：98 个 NexusPHP 站点中仅 64 个支持 pieces_hash API。加上 10 个非 NP 站点（全部不支持），有 **44 个站点无法通过 pieces_hash 精确辅种**。

### 1.2 核心矛盾

对 L1.5/L3（file_tree/fingerprint），匹配前提是 DB 中有目标站的指纹数据。但指纹数据来自 .torrent 文件，而对一个新目标站，下载器中通常没有该站种子——如果有，就不需要辅了。这形成了**循环依赖**：

```
需要指纹数据 → 需要下载 .torrent → 需要知道哪些种子存在 → 需要匹配（回到起点）
```

### 1.3 竞品方案对比

| 竞品 | 无 API 站点方案 | 是否需要第三方依赖 |
|------|---------------|-------------------|
| **cross-seed** | Torznab 标题搜索 → 下载 .torrent → 文件树比对 | 需要 Jackett/Prowlarr |
| **Graft** | 本地内容指纹匹配 | 需要预采集目标站数据 |
| **Reseed-backend** | 文件名+大小索引 | 需要预采集 |
| **IYUUPlus** | 云端 info_hash 匹配 | 需要 IYUU 云服务 |

### 1.4 设计目标

1. **不引入第三方依赖**（Jackett/Torznab/IYUU 等）
2. **利用现有基础设施**：所有框架的 adapter 已实现 `SearchTorrents`
3. **覆盖全框架**：NexusPHP、Unit3D、Gazelle、MTeam、TNode、Rousi、Yemapt、Generic
4. **尊重站点流控**：搜索频控、搜索缓存

## 2. 设计方案：搜索验证匹配（Search-Verify Matching）

### 2.1 核心流程

```
┌─────────────────────────────────────────────────────────────────────┐
│ L2 搜索验证匹配流程                                                  │
│                                                                     │
│  源种子（已知：标题、大小）                                            │
│      │                                                              │
│      ▼                                                              │
│  ① 标题截断：去中文前缀 + 截到分辨率关键词                             │
│      │ 无分辨率 → 跳过                                               │
│      ▼                                                              │
│  ② 提取小组名：取最后一个 - 之后的部分（去文件后缀）                    │
│      │ 无小组名 → 跳过                                               │
│      ▼                                                              │
│  ③ 搜索缓存命中？                                                    │
│      │ 命中 → 用缓存结果（跳过搜索，节省配额）                          │
│      │ 未命中 ↓                                                      │
│      ▼                                                              │
│  ④ SearchTorrents(截断标题) → 候选列表（标题+大小+ID）                 │
│      │ 搜索失败 → 返回 nil                                           │
│      ▼                                                              │
│  ⑤ 小组名过滤：只保留标题含小组名的候选                                │
│      │ 无匹配 → 返回 nil                                             │
│      ▼                                                              │
│  ⑥ 大小精确匹配（字节级）：找到大小完全一致的候选                       │
│      │ 无匹配 → 返回 nil                                             │
│      ▼                                                              │
│  ⑦ 匹配成功 → 返回 Candidate（confidence=0.95, MatchMethod=search_verify）│
└─────────────────────────────────────────────────────────────────────┘
```

**关键**：全程不需要下载 .torrent 文件，只消耗搜索配额。三步匹配（截断标题+小组名+精确大小）精度等同于文件树比对。

### 2.2 与现有匹配层的关系

L2 搜索验证是现有 L2 size_title 的**升级替代**。新的匹配层优先级：

```
L0 pieces_hash（API 远端匹配，零配额消耗）
  ↓ miss
L1.5 file_tree（本地 DB 匹配，需要已有指纹数据）
  ↓ miss
L2 搜索验证（标题截断搜索 → 小组名过滤 → 大小精确匹配）
  ↓ miss
L3 fingerprint（本地 DB 匹配，兜底）
```

### 2.3 现有基础设施支持

所有框架的 adapter 均已实现 `SearchTorrents`：

| 框架 | 站点数 | SearchTorrents | 说明 |
|------|--------|---------------|------|
| NexusPHP | 97 | 有 | NP 搜索页解析 |
| Generic | 2 | 有 | HTML 搜索页解析（Unit3D/Gazelle/TNode/Yemapt 继承） |
| MTeam | 1 | 有 | API 搜索 |
| Rousi | 1 | 有 | 自有搜索接口 |

## 3. 标题截断规则（ExtractSearchKeyword）

### 3.1 步骤

**Step 1：去除中文前缀**

识别标题开头到第一个英文字母/数字之间的中文字符（含 `[中文].` 格式），整体去除。

- `三岛由纪夫传.Mishima...` → 去除 `三岛由纪夫传.`
- `[最好的我们].My.Best...` → 去除 `[最好的我们].`
- `[国语].骗骗喜欢你.Honey...` → 去除 `[国语].`
- `The.Matrix.1999...` → 无中文前缀，不去除

**Step 2：截取到分辨率关键词**

从 Step 1 结果中，找到第一个分辨率关键词，截取包含该关键词在内的前半部分。

分辨率关键词列表：`2160p`、`1080p`、`720p`、`480p`（大小写不敏感）

- 无分辨率关键词 → 返回空（跳过，不搜索）

## 4. 小组名提取规则（ExtractGroupName）

### 4.1 步骤

1. 取标题中**最后一个** `-` 之后的部分
2. 去除文件后缀（`.mkv`、`.mp4`、`.avi`、`.ts` 等）
3. 结果即为小组名

### 4.2 规则

- PT 站点资源必须有明确小组来源，末尾必须有 `-GROUP` 格式
- 如果没有 `-GROUP` 格式 → 返回空（跳过，不搜索）
- 特殊小组名：`NOGROUP`、`NoGroup` 等也是合法的

### 4.3 提取示例

| 标题末尾 | 小组名 |
|---------|--------|
| `MNHD-FRDS` | `FRDS` |
| `mUHD-FRDS` | `FRDS` |
| `DDP5.1-HHWEB` | `HHWEB` |
| `Mandarin-CSWEB` | `CSWEB` |
| `AAC-UBWEB.mkv` | `UBWEB` |
| `WEB-NOGROUP` | `NOGROUP` |
| `WEB-NoGroup` | `NoGroup` |

## 5. 三步精确匹配

### 5.1 第一步：标题截断搜索

用 `ExtractSearchKeyword` 截断后的标题调用 `SearchTorrents`，获取候选列表。

### 5.2 第二步：小组名过滤

用 `ExtractGroupName` 提取源种子的小组名，在搜索结果中只保留标题包含该小组名的候选。

**必须匹配**：搜索结果的标题中必须包含源种子的小组名字符串。

### 5.3 第三步：大小匹配

由于站点搜索结果页面以 `GB`/`MB` 格式展示大小（如 `12.37 GB`），解析回字节存在精度损失，无法做字节级精确比对。因此将两端统一转换到同一单位后比较：

**规则：**
- `≥ 1 GB`：两端均转为 GB（2 位小数）后比较
- `< 1 GB`：两端均转为 MB（2 位小数）后比较

**示例：**
```
源种子:  13,280,258,459 bytes → round(/ 1024³, 2) = 12.37 GB
搜索结果: "12.37 GB" → parseSizeStr → round(/ 1024³, 2) = 12.37 GB
→ 匹配

源种子:  537,522,046 bytes → round(/ 1024², 2) = 512.55 MB
搜索结果: "512.55 MB" → parseSizeStr → round(/ 1024², 2) = 512.55 MB
→ 匹配
```

匹配成功 → 返回 Candidate（confidence=0.95, MatchMethod=`search_verify`）

### 5.4 完整匹配流程示例

```
源种子标题: 三岛由纪夫传.Mishima.A.Life.in.Four.Chapters.1985.UHD.BluRay.1080p.x265.10bit.FLAC.MNHD-FRDS
源种子大小: 13,280,258,459 bytes

① 截断搜索词: Mishima.A.Life.in.Four.Chapters.1985.UHD.BluRay.1080p
② 提取小组名: FRDS
③ SearchTorrents("Mishima.A.Life.in.Four.Chapters.1985.UHD.BluRay.1080p") → 候选列表（~20条）
④ 小组名过滤: 只保留标题含 "FRDS" 的结果 → 剩 ~3条
⑤ 大小匹配: 找到 13,280,258,459 bytes 的候选 → 确认匹配
⑥ 返回 Candidate{TargetSite, TargetTorrentID, Confidence: 0.95}
```

### 5.5 为什么不需要下载 .torrent 做文件树比对

三步匹配（截断标题搜索 + 小组名 + 精确大小）已经足够精确：
- 标题+分辨率 → 锁定具体发行版本
- 小组名 → 锁定同一制作组
- 精确字节大小 → 排除所有误判可能

**唯一的理论风险**：两个不同资源恰好标题、小组名、字节大小完全相同 → 实际 PT 环境中概率极低（不同资源总有至少 1 字节差异）。

**优势**：不消耗 .torrent 下载配额，不增加站点负担。

### 5.6 异常处理

| 异常情况 | 处理方式 |
|---------|---------|
| 无分辨率关键词 | 跳过，不搜索 |
| 无小组名（末尾无 `-XXX`） | 跳过，不搜索 |
| 搜索返回空结果 | 记录日志，继续下一个源种子 |
| 搜索失败（网络/限流） | 记录日志，继续下一个源种子 |
| 小组名过滤后无候选 | 记录日志，继续下一个源种子 |
| 大小匹配后无候选 | 记录日志，继续下一个源种子 |
| 大小匹配后多个候选（理论上不应发生） | 取第一个 |

## 6. 搜索缓存

### 6.1 search_result_cache 表

| 字段 | 类型 | 说明 |
|------|------|------|
| site_name | TEXT | 目标站点名 |
| keyword | TEXT | 截断后的搜索关键词 |
| results | TEXT | JSON 序列化的 `[]SeedingSearchResult` |
| has_match | BOOL | 是否有匹配结果 |
| created_at | DATETIME | 缓存时间 |
| expires_at | DATETIME | 过期时间（默认 24h，可配） |

UNIQUE 索引：`(site_name, keyword)`

**作用**：
- 有结果的缓存：同一关键词不重复搜索，存于数据库可做统计分析
- 无结果的缓存：记录下来，下次运行时重新匹配（今天没有不代表明天没有）
- 缓存 TTL 由辅种任务定时器控制（用户可配置运行周期）

**搜索关键词去重**：同一截断标题对同一目标站只搜一次，多源种子共享同一搜索结果。

**容量估算**：~6000 个唯一截断标题 × 目标站数，每条 ~2KB JSON，可接受。

### 6.2 缓存淘汰

| 缓存类型 | 淘汰策略 |
|---------|---------|
| 搜索结果（有匹配） | TTL 过期（默认 24h，可配），过期后重新搜索 |
| 搜索结果（无匹配） | TTL 过期（默认 24h，可配），过期后重新搜索（站内可能新增种子） |

### 6.3 confirmedTargets 联动

L0 pieces_hash 已确认匹配的目标种子（`confirmedTargets`），L2 也必须跳过，避免重复注入。复用现有的 `confirmedTargets` 机制。

## 7. 流控设计

### 7.1 站点配额模型

```
┌────────────────────────────────────────────┐
│          站点配额（site 配置）                │
│                                            │
│  search_interval_s: 3    (搜索最小间隔)      │
│  search_daily_limit: 0   (每日搜索上限，0=不限)│
│                                            │
│          运行时计数器（DB）                    │
│                                            │
│  searches_today: 45      (今日已搜索次数)     │
│  last_search_at: 18:30   (上次搜索时间)       │
└────────────────────────────────────────────┘
```

注：三步匹配不需要下载 .torrent 文件，只需控制搜索配额。

### 7.2 配额存储

新增 `reseed_site_quota` 表：

| 字段 | 类型 | 说明 |
|------|------|------|
| task_id | INT | 辅种任务 ID |
| site_name | TEXT | 站点名 |
| date | TEXT | 日期（YYYY-MM-DD） |
| search_count | INT | 当日搜索次数 |
| last_search_at | DATETIME | 上次搜索时间 |

每日零点重置（或按 date 字段自动判断）。

### 7.3 搜索并发模型

多目标站点时**并行搜索**，单站串行（遵守 search_interval_s）：

```
目标站A: keyword1 → (3s) → keyword2 → (3s) → keyword3 ...
目标站B: keyword1 → (3s) → keyword2 → (3s) → keyword3 ...
  ↑ 并行执行
```

单站 6000 个关键词 × 3s = ~5 小时（首次全量）。后续运行靠缓存，只搜新增关键词。

首次全量扫描的耗时由用户通过以下方式控制：
- `search_interval_s`：搜索间隔（默认 3s，可调低）
- `search_daily_limit`：每日搜索上限（0=不限）
- 任务定时器周期：用户可配置运行频率

缓存跨任务持久化，不会因任务结束而丢失。几轮运行后自然覆盖全部关键词。

### 7.4 前端配置

站点设置页新增字段：

| 字段 | 默认值 | 说明 |
|------|--------|------|
| search_interval_s | 3 | 搜索最小间隔（秒） |
| search_daily_limit | 0 | 每日搜索上限（0=不限） |

## 9. 风险与缓解

| 风险 | 概率 | 缓解措施 |
|------|------|---------|
| 站点搜索接口限制严格 | 中 | 搜索缓存（24h TTL）+ 每日上限 |
| 标题截断不准导致搜不到 | 中 | 截断规则遵循 PT 通用标题规范，覆盖常见格式 |
| 站点搜索返回 HTML 格式不一致 | 低 | 各 adapter 已实现 `SearchTorrents`，框架层面已适配 |
| 不同站点标题格式差异（如空格 vs 点号） | 中 | 搜索关键词保留点号，NexusPHP 搜索通常支持点号分隔 |
| 同一资源不同站小组名不同 | 低 | 取最后一个 `-` 后的部分，不同站的小组名通常一致 |

## 10. 决策记录

| 决策 | 选择 | 理由 |
|------|------|------|
| 不引入 Jackett | 复用现有 adapter | adapter 已覆盖全部框架，无额外依赖 |
| 不下载 .torrent 验证 | 三步精确匹配（标题+小组名+大小） | 不消耗下载配额，PT 标题规范保证唯一性 |
| 搜索缓存 | 24h TTL | 每日最多搜一次同一关键词 |
| L2 替代而非新增 Layer | 保持 4 层架构 | L2 本就是标题搜索，升级为精确验证 |
| 无分辨率/无小组名跳过 | 不搜索 | 不符合 PT 标题规范的资源不处理 |
 | 大小按 GB/MB（2 位小数）匹配 | 不做字节级比较 | 站点搜索结果精度为 GB/MB 级，字节级比较不可行 |
| 配额按站点+任务维度管理 | 不同站限制不同 | 灵活适配各站流控策略 |
