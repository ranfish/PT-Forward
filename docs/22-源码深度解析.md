## 1. 辅种引擎核心算法对比

### 1.1 六大引擎架构概览

#### ptdog (Go) — 本地 pieces_hash 匹配引擎

```
┌─────────────────────────────────────────────────────────────┐
│                    Reseed.Run() 主流程                       │
├─────────────────────────────────────────────────────────────┤
│  Scanner(扫描本地) → Querier(查询站点) → Seeder(执行辅种)    │
│         │                  │                   │             │
│  .torrent文件       pieces_hash          skip_checking      │
│  → info_hash        批量查询API          + save_path        │
│  → pieces_hash                                               │
│                                                              │
│  特点: 纯本地计算 · pieces_hash跨站一致 · 需站点API支持      │
└─────────────────────────────────────────────────────────────┘
```

```go
type Reseed struct{}
func (r *Reseed) Run() error {

    querier.Websites(websites).Run()

}
```

Scanner 核心逻辑 ([scanner.go:44-73](file:///home/incast/PT-Forward/examples/ptdog/app/reseed/scanner.go)):
```go
func (s *Scanner) torrents() (map[string]*client.Torrent, error) {
    hashes, _ := s.load()          // map[info_hash]pieces_hash
    data, _ := s.client.Torrents(queries)
    for _, t := range data {
        if !t.IsFinished { continue }
        if piecesHash, ok := hashes[t.InfoHash]; ok {
            t.PiecesHash = piecesHash
            torrents[t.PiecesHash] = t
        }
    }
}
```

**关键创新**: 使用 `pieces_hash`（分片哈希）而非 `info_hash` 作为跨站匹配键。

### 1.2 匹配算法深度对比

| 维度 | **ptdog** | **cross-seed** | **Graft** | **iyuuplus** | **reseed-puppy** | **Reseed-backend** |
|------|-----------|----------------|-----------|--------------|------------------|--------------------|
| **匹配键** | pieces_hash | title+files+size | ContentFingerprint | info_hash | size(GB级) | file×size(±5%) |
| **匹配方式** | API查询 | 本地内容比对 | 本地分层指纹 | 云API查询 | Jackett搜索 | 本地DB查询 |
| **依赖外部** | 需站点API | 不需要 | 不需要 | IYUU云 | Jackett | 集中DB |
| **误判率** | ≈0% | ~3% | ~1% | ≈0% | 较高 | 低 |
| **站点覆盖** | 受限API | 任意Torznab | 任意 | 123站 | Jackett覆盖 | DB录入量 |
| **离线能力** | 部分 | 完全 | 完全 | 无 | 部分 | 完全 |

**准确性排名**: ptdog ≈ iyuuplus > Graft > Reseed-backend > cross-seed > reseed-puppy
**灵活性排名**: cross-seed > Graft > Reseed-backend > iyuuplus > ptdog > reseed-puppy
**易用性排名**: iyuuplus > ptdog > cross-seed > reseed-puppy > Graft > Reseed-backend

### 1.3 PT-Forward三层融合策略建议

基于六大引擎分析，PT-Forward应采用三层融合：
- **Layer 1 (本地快速)**: pieces_hash精确匹配（ptdog方案），O(1)查找
- **Layer 2 (结构化)**: ContentFingerprint分层匹配（Graft方案），±5%容差
- **Layer 3 (搜索引擎)**: 文件树比对+大小匹配（cross-seed方案），最大覆盖面

### 2.2 rflush 刷流引擎

**架构**: Rust + Tokio + Axum + SQLite + React

**核心亮点: 两轮过滤设计** ([scheduler.rs](file:///home/incast/PT-Forward/examples/rflush/src/brush/scheduler.rs)):
```
第一轮 (RssPreFilter): 仅用RSS属性快速筛选，缺少promotion/H&R时不提前淘汰
第二轮 (PostEnhancement): 仅对候选种子请求站点详情页，用完整属性严格筛选
```

**好处**: 大幅减少站点HTTP请求，避免限流，同时确保不遗漏免费种子。

选种规则:
| 规则 | 说明 |
|------|------|
| 体积/做种人数范围 | JSON `["1-10","20-50"]` GB |
| 促销类型 | all/free/normal |
| 活跃时间窗口 | `["00:00-09:00","22:00-06:00"]` 支持跨天 |
| 最小免费时长 | 仅promotion=free时生效 |

删种规则 (8种 + OR/AND模式):
| 规则 | 说明 |
|------|------|
| 最小做种时长 | 做种时间达标后可删 |
| free到期删除 | 促销到期后删除 |
| H&R最小做种 | 仅对H&R种子 |
| 目标分享率 | 分享率达标后可删 |
| 最低平均上传 | 近10分钟平均上传低于阈值 |
| 最大不活跃 | 最后活动时间超时 |

2. 站点级差异（CSS选择器/字段映射/认证方式）通过声明式数据定义
3. 发布级差异（表单字段/API端点）通过YAML映射

## 7. Cookie与认证安全

### 7.1 CookieCloud — 端到端加密Cookie同步

**双模式AES加密架构**:

| 模式 | 密钥派生 | 算法 | IV | 安全性 |
|------|---------|------|-----|--------|
| legacy | MD5(uuid-password)[:16] | OpenSSL EVP_BytesToKey + AES-256-CBC | 随机salt | 低 |
| aes-128-cbc-fixed | MD5(uuid-password)[:16] | AES-128-CBC | 全零固定 | 中 |

**核心原则**: 服务端只存储密文，永远不接触明文Cookie。

多语言解密支持: Go/Python/Java/PHP/Node/Deno 六种语言实现。

## 8. 架构模式提炼

### 8.1 设计模式总结

| 模式 | 使用项目 | 应用场景 |
|------|---------|---------|
| **Driver泛型** | pt-tools | `Driver[Req,Res]`分离请求准备/执行/解析 |
| **三层映射** | PTNexus | source_parsers → standard_keys → mappings |
| **声明式定义** | pt-tools | SiteDefinition数据驱动，零代码新增站点 |
| **策略模式** | 几乎所有 | 站点适配器/下载器/图床的统一接口+多实现 |
| **工厂模式** | ARSS/ADTU/pt-tools | 动态创建站点适配器实例 |
| **生产者-消费者** | ARSS/rflush | RSS定时轮询→队列缓冲→过滤后下载 |
| **模板方法** | PTNexus/Graft | publicSitePublisher可覆写步骤 |
| **回退模式** | PTNexus | 特殊提取器失败→回退通用提取器 |
| **熔断器** | pt-tools | 三态(Closed/Open/Half-Open)站点保护 |
| **文件自动发现** | Vertex | require()动态加载站点适配器 |
| **规则引擎** | Vertex/rflush | 条件组合+eval/声明式双模式 |
| **事件驱动** | iyuuplus | Event::emit(before/after/done)生命周期钩子 |

### 8.2 反模式警示

| 反模式 | 项目 | 严重程度 | 说明 |
|--------|------|---------|------|
| **eval()执行用户代码** | Vertex/reseed-puppy/Reseed-backend | 🔴 严重 | 安全漏洞，应使用沙箱DSL |
| **30K行单文件** | auto_feed | 🔴 严重 | 无法维护/测试/增量更新 |
| **全局状态泛滥** | Vertex (30+ global.*) | 🟠 高 | 影响测试和可维护性 |
| **每次API调用都重新登录** | ptdog/harvest_rust | 🟡 中 | 应缓存SID/Session |
| **代码编译闭源** | ARSS/ADTU | 🟡 中 | 阻碍社区贡献和审计 |
| **大小匹配精度过低** | reseed-puppy (GB级四舍五入) | 🟡 中 | 高误判率 |
| **全零固定IV** | CookieCloud | 🟡 中 | 相同key+明文→相同密文 |
| **同步文件I/O** | Vertex (readFileSync) | 🟡 中 | 阻塞事件循环 |
| **map[string]any滥用** | PTNexus | 🟢 低 | 缺乏类型安全 |
| **bare except** | screenshot | 🟢 低 | 吞掉所有异常 |

### 10.1 架构选型

| 模块 | 借鉴来源 | 方案 |

| **限流** | rflush | FIFO滑动窗口 + 域名级自动冻结 |

### 10.2 必须避免的陷阱

1. **绝不使用eval()** — 用声明式规则DSL替代
2. **绝不编译闭源** — 开源社区驱动
3. **绝不30K单文件** — 模块化架构
4. **绝不GB级大小匹配** — 字节级精确比对
5. **绝不每次重新登录** — 缓存SID/Session
6. **绝不全局状态泛滥** — 依赖注入
7. **绝不全零固定IV** — 使用随机IV + PBKDF2密钥派生
8. **绝不裸SQL拼接** — 参数化查询
